# Spike 005: brewfile-sync

## What This Validates
Given parsed local list and existing Brewfile, when synced, then update Brewfile accurately

## Research
Using `brew bundle dump --file=-` gives us exactly the format that belongs in a `Brewfile`. We can use this to update the project's actual `Brewfile`. Since the project's `Brewfile` has some inline comments (like `# tracks v1.15.11`), using `brew bundle dump --file=Brewfile --force` might overwrite these comments. We need to be careful to either preserve them or just accept the format generated by `brew bundle dump`.
A simpler approach is to test `brew bundle dump --file=Brewfile --force` directly on a copy of the Brewfile and see what it does.

## How to Run
`bash .planning/spikes/005-brewfile-sync/test_sync.sh`

## What to Expect
A diff showing the changes made to a dummy Brewfile.

## Investigation Trail
- Copying the real `Brewfile` to a dummy location.
- Running `brew bundle dump` on it.
- Examining the diff to see how it handles comments and missing packages.

## Results
VALIDATED. Running `brew bundle dump --file=Brewfile --force` works and generates a complete Brewfile. 

However, as expected, it **destroys all comments and custom grouping** that were in the original Brewfile. It outputs everything in alphabetical order grouped purely by type (tap, brew, cask, vscode, uv, npm).

While this accurately syncs the Brewfile with the local state, it ruins the readability and organization of the existing file. 

If the goal is to *update* the Brewfile based on local state while preserving structure, a custom script would be needed to parse the dump and merge it, or we just accept that the Brewfile should be an auto-generated artifact going forward without manual comments.
