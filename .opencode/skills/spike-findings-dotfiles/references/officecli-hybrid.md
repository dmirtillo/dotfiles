# OfficeCLI Hybrid Pattern

## Requirements

- The client MUST explicitly request the MCP tools in its API call using LiteLLM's custom `{"type": "mcp", "server_url": "litellm_proxy"}` syntax.
- Clients like OpenCode and Gemini CLI cannot "implicitly" inherit tools configured on the LiteLLM proxy without code modifications.
- **Markitdown+OfficeCLI:** Must explicitly call `officecli close <file>` to flush edits to disk before passing the file to `markitdown` for reading, due to `officecli`'s resident mode.

## How to Build It

When working with Office Documents (.docx, .pptx, .xlsx) inside GSD pipelines or AI orchestration:

1. **Read Strategy:**
   Use `markitdown` with the `[all]` extra to read all documents. This is installed globally via `uv tool install 'markitdown[all]'`.
   ```bash
   uvx --with 'markitdown[all]' markitdown presentation.pptx
   ```

2. **Write Strategy:**
   Use `officecli` directly for all DOM injections and text-matching replacement.
   ```bash
   officecli set document.docx --find "DRAFT" --replace "FINAL"
   ```

3. **The Flush (Critical):**
   Between any write operation and a subsequent read operation, the in-memory Resident Mode daemon MUST be explicitly flushed to disk, or `markitdown` will extract stale XML.
   ```bash
   officecli close document.docx
   ```

## What to Avoid

- **Avoid LLM Markdown-to-DOM Parsing:** Do not attempt to parse raw Markdown formatting diffs generated by an LLM into OOXML structures. It is highly complex and fails on lists/tables due to relational `numId` schemas. Rely entirely on targeted `--find`/`--replace` and explicit DOM `add` operations.
- **Avoid Internal Reads:** Do not use `officecli view` or `officecli get` as standard read tools. The AI prompt should explicitly strip these out and demand `markitdown`.
- **Avoid Multimodal OCR on Large Decks:** MarkItDown processes OCR sequentially. Running `markitdown` with the OCR plugin on a file with 50+ images will trigger a 150+ second freeze, violating the standard 120s CLI tool execution timeout. Run without OCR first, then target specific images if necessary.

## Constraints

- Concurrent background writes (`officecli add ... &`) are perfectly safe and queued via the Resident Mode daemon without corruption.
