# Spike 038: Markdown to DOM Translation

## What This Validates
Given standard markdown diffs from an LLM, when parsed, then can they be reliably translated into `officecli` DOM or text-replace operations.

## How to Run
```bash
python3 .planning/spikes/038-markdown-to-dom-translation/parse_test.py
```

## Investigation Trail
The `markitdown` output contains HTML comments like `<!-- Slide number: 1 -->`. If an LLM produces a unified diff targeting a piece of text inside that slide, mapping that text back to a specific `/slide[1]/shape[N]` XML DOM path requires full AST parsing of the Markdown or heuristic regex. 

However, `officecli`'s text-replace capability (`officecli set <file> / --prop find="..." --prop replace="..."`) operates globally across the entire XML DOM tree and inherently supports the identical plain-text strings generated by `markitdown`.

## Results
⚠ PARTIAL.
Translating Markdown back to explicit DOM paths (`/slide[2]/shape[4]`) is unreliable and fragile. However, this translation isn't actually necessary. Standard text diffs from LLMs can be piped directly into `officecli set <file> / --prop find="old" --prop replace="new"`, bypassing the need for DOM mapping entirely.
