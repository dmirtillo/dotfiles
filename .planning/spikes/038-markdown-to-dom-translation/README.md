# Spike 038: Markdown to DOM Translation

## What This Validates
Can we write a simple parser that takes standard Markdown diffs/edits generated by an LLM and applies them to an office document via `officecli`?

## Research
Currently, LLMs use bash commands (`officecli set ... --prop find=old --prop replace=new`) to edit documents. If the LLM generates a large block of markdown representing the new state of a slide or document section, translating that directly requires either mapping markdown syntax back to DOM calls (`add --type paragraph --prop text="..." --prop bold=true`) or just using heavy text replacement.

## How to Run
```bash
# Provide a markdown snippet
# Attempt to parse it into an officecli batch JSON or sequence of commands
```

## What to Expect
A Python script that reads markdown and converts standard elements (headers, lists, bold text) into an `officecli batch` JSON.

## Investigation Trail
1. Creating a parser from markdown AST to OpenXML DOM via `officecli` batch operations is essentially building a full markdown-to-OOXML engine. `officecli` handles the OOXML part well, but mapping markdown blocks (like parsing a list item with inline bold) into precise `officecli add` calls (e.g., adding a paragraph with multiple runs) is highly complex.
2. `officecli` does not natively accept raw markdown as an input type yet (e.g., `officecli add /body --type markdown --prop text="**Bold**"`).
3. If an LLM needs to make extensive edits, the current text-replacement mechanism (`officecli set / --prop find="..." --prop replace="..."`) works, but recreating full document structures from markdown would be extremely brittle.

## Results
**INVALIDATED ✗**
While standard text and headers (`Heading1`) can be mapped somewhat trivially into a JSON batch array for `officecli`, fully translating Markdown features like lists, tables, and nested formatting into OOXML DOM commands is highly complex. For example, simply setting the `style="ListParagraph"` does not natively render a proper bulleted list in OOXML—it requires specific `numId` and `ilvl` properties tied to the document's `numbering.xml` definitions. 

Building an intermediary markdown-to-DOM parser is an anti-pattern. Instead, the LLM should continue using explicit targeted `add` or `set --find --replace` commands via `officecli` to manipulate specific content reliably.
